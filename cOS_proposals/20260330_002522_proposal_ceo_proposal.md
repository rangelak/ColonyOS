## Proposal: Codebase Context Map for Phase Prompts

### Rationale
Every phase (plan, implement, review) currently relies on Claude's in-session exploration to discover codebase structure — burning tokens and context window on `find`, `grep`, and file reads that rediscover the same architecture every run. A pre-computed structural map injected into phase prompts would give agents instant architectural awareness, reduce wasted exploration tokens, and dramatically improve plan quality on larger codebases. This is the #1 technical gap identified in the strategic directions and the key blocker for scaling beyond small-to-medium repos.

### Builds Upon
- "Persistent Memory System" (memory injection into phase prompts provides the pattern)
- "Intent Router Agent" (complexity classification benefits from knowing codebase size/shape)
- "Cross-Run Learnings System" (same prompt-injection architecture)

### Inspired By
Aider's repo-map concept (cited as a critical gap in STRATEGIC_DIRECTIONS.md under "Context Window Exhaustion") and Superpowers' "subagent-per-task with fresh context" pattern — fresh context is only effective if agents start with structural awareness rather than spending half their budget on orientation.

### Feature Request
Build a **codebase context map** system that generates a lightweight structural index of the repository and injects it into phase prompts, giving every agent instant architectural awareness without exploration overhead.

**Core module (`src/colonyos/repomap.py`):**
- Use Python's `ast` module to extract classes, functions, and top-level imports from `.py` files (zero new dependencies)
- Use simple regex patterns for JS/TS (exported functions/classes), Go (func/type declarations), and Rust (pub fn/struct) — cover the common cases
- Generate a hierarchical structure: `{path: {classes: [...], functions: [...], imports: [...]}}`
- Respect `.gitignore` patterns and a configurable `exclude_patterns` list (e.g., `node_modules/`, `vendor/`, `__pycache__/`)
- Output stored as `.colonyos/repo-map.json` (machine-readable) with a human-readable summary renderer
- Include a staleness check: store a hash of the file tree; regenerate only when files have changed

**Prompt injection:**
- Add a `_build_repo_context()` method in the orchestrator that loads the repo map and renders a condensed summary (e.g., "src/colonyos/orchestrator.py: classes [Orchestrator], functions [run, run_ceo, _run_pipeline, ...], 4400 lines")
- Inject into PLAN, IMPLEMENT, and REVIEW phase prompts as a `{repo_map}` template variable
- Configurable `max_map_tokens` (default ~2000) that truncates intelligently — prioritize files most relevant to the current task by matching task keywords against file paths and symbol names
- For IMPLEMENT phases, narrow the map to files relevant to the current task (keyword matching against the task description)

**CLI command:**
- `colonyos index` — manually generate/regenerate the repo map with `--verbose` flag to show what was indexed
- `colonyos index --stats` — show summary: N files indexed, N symbols extracted, map size in tokens

**Configuration (`RepoMapConfig` dataclass):**
- `enabled: bool = True`
- `max_map_tokens: int = 2000`
- `exclude_patterns: list[str] = ["**/node_modules/**", "**/__pycache__/**", "**/venv/**"]`
- `auto_refresh: bool = True` (regenerate at pipeline start if stale)
- `languages: list[str] = ["python", "javascript", "typescript", "go", "rust"]`

**Pipeline integration:**
- At pipeline start (after preflight, before plan), check if repo map exists and is fresh; regenerate if needed
- The repo map is NOT generated by an LLM — it's pure static analysis, so it's free and instant
- Add `.colonyos/repo-map.json` to `.gitignore`

**Acceptance criteria:**
1. `colonyos index` generates a valid repo-map.json for a Python project with correct class/function extraction
2. PLAN phase prompts include the repo map summary when enabled
3. IMPLEMENT phase prompts include a task-relevant subset of the map
4. Repo map auto-regenerates when file tree changes between runs
5. `RepoMapConfig` is parsed from config.yaml with sensible defaults
6. Map respects `.gitignore` and `exclude_patterns`
7. Unit tests cover: AST extraction, staleness detection, token truncation, keyword relevance filtering, config parsing
8. Works correctly on the ColonyOS repo itself (dogfooding)
