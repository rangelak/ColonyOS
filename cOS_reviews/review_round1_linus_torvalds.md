# Review by Linus Torvalds (Round 1)

I've now reviewed the entire implementation. Here's my review:

---

VERDICT: request-changes

FINDINGS:
- [src/colonyos/cli.py:1137-1145]: **FR-5.2 approval gate is a lie.** The code posts "Awaiting approval — react with :thumbsup: to proceed" but then immediately proceeds to run the pipeline on the very next line without actually waiting for any reaction. This is not a feature — this is a deception. The user sees "awaiting approval" while the pipeline is already executing. Either implement the gate or remove the message. This was the ONE security concession the PRD made to get the security-concerned reviewers on board.
- [src/colonyos/cli.py:1087-1185]: **FR-3.2 emoji reaction triggers not implemented.** The PRD explicitly requires listening for emoji reactions (`reaction_added` events) as an alternative trigger mechanism. Only `app_mention` is registered. The `trigger_mode` config field accepts "reaction" as a value but there's no handler for it — a config knob that does nothing is worse than not having one.
- [src/colonyos/cli.py:1071-1072]: **`effective_max_hours` and `effective_max_budget` are computed then never enforced.** The auto command checks these in its loop. The watch command computes them into local variables that are never referenced again. Dead code. The watcher will run forever and spend unlimited money regardless of what you configure.
- [src/colonyos/cli.py:1130-1134]: **Dedup mark happens before pipeline runs.** `mark_processed` is called at the top of `_run_pipeline()`, before the orchestrator is invoked. If the pipeline crashes, the message is permanently marked as processed and can never be retried. The mark should happen after successful completion, or there should be a "pending" vs "completed" state.
- [src/colonyos/cli.py:1182]: **Unbounded concurrent pipeline threads.** Each mention spawns a new daemon thread calling `run_orchestrator` which does git operations. Two simultaneous mentions will race on the same working tree — corrupting state, conflicting branches, partial commits. There's no semaphore, no queue, nothing. The PRD flagged this as an open question; the implementation ignores it entirely and just hopes for the best.
- [src/colonyos/slack.py:33]: **Duplicated `_XML_TAG_RE` regex.** The PRD explicitly says to extract `_sanitize_untrusted_content` from `github.py` for reuse or move to a shared module. Instead, the regex is copy-pasted. Now you have two copies to keep in sync, and they will diverge.
- [src/colonyos/cli.py:1050]: **`SlackUI` is imported but never used.** The import exists but `run_orchestrator()` is called without passing any SlackUI instance. Phase updates don't go to Slack threads — they go to the terminal (or nowhere). FR-6.3 (phase completion updates in thread) is not actually working.
- [src/colonyos/cli.py, src/colonyos/ui.py, tests/test_cli.py, tests/test_registry_sync.py]: **Unrelated changes.** This branch includes a full interactive REPL mode, markdown rendering in the terminal UI, dynamic banner generation, and registry sync tests. None of this is in the Slack integration PRD. These should be on their own branch. This is sloppy branch discipline.
- [src/colonyos/slack.py:365-395]: **`create_slack_app` ignores the config parameter.** The function takes `config: SlackConfig` but never reads any field from it. The channels, trigger_mode, allowed_user_ids — none of it is used to configure the Bolt app. The config is just decoration.
- [src/colonyos/cli.py:1174]: **Bare `except Exception` with only logging.** If the pipeline fails, the error is logged but the Slack thread gets no error details — just an ❌ emoji. FR-6.5 says to react with ❌ on failure (done) but there's no error summary posted to the thread explaining what went wrong.

SYNTHESIS:
This implementation gets the easy parts right — config parsing, content sanitization, message filtering, test coverage for utility functions — and completely botches the hard parts. The approval gate is theater: it tells the user it's waiting for approval while already executing. The reaction trigger mechanism is configured but not wired up. Budget and time limits are computed into variables that nobody reads. Concurrent pipelines will corrupt git state. The `SlackUI` class is built but never plugged in, so phase updates don't actually reach Slack threads. And the branch is polluted with an entire REPL feature that has nothing to do with Slack integration.

The data structures are fine. The sanitization is fine. The tests pass — but they test the *easy* things (string formatting, dict serialization) and don't test the *hard* things (the actual event handler behavior, the approval gate, concurrency). What's here is a demo, not a feature. Fix the approval gate, wire up reaction triggers, enforce the budget limits, serialize pipeline execution, plug in SlackUI, and split out the unrelated REPL changes.
