# Review by Linus Torvalds (Round 1)

Here's my review:

---

VERDICT: request-changes

FINDINGS:
- [Formula/colonyos.rb:10]: **PLACEHOLDER_SHA256** — This is a literal placeholder string shipped as production code. The formula is non-functional. You can't ship a Homebrew formula with `sha256 "PLACEHOLDER_SHA256"`. Either compute the real checksum or don't ship the formula at all. A file that doesn't work is worse than no file, because it looks like it works until someone actually tries it.
- [Formula/colonyos.rb:9]: **Hardcoded version URL** — `colonyos-0.1.0.tar.gz` is hardcoded, which directly contradicts the entire point of FR-3 (single-source versioning). The moment you push `v0.2.0`, this formula points at the old tarball. The PRD's FR-5.4 requires the release workflow to auto-update the formula — that step is completely missing from `release.yml`.
- [.github/workflows/release.yml]: **No Homebrew formula update step** — FR-5.4 explicitly requires "Auto-update the formula on each release (via the release workflow)." The release workflow has zero mention of the Homebrew formula. The formula is dead-on-arrival for any release after v0.1.0.
- [.github/workflows/release.yml]: **Duplicated test job** — The entire test job (lines 72-95) is copy-pasted from `ci.yml`. This is the kind of lazy duplication that rots. Use a reusable workflow (`workflow_call`) or at minimum a composite action. When someone changes the test matrix in `ci.yml` and forgets `release.yml`, you get silent divergence. I've seen this kill projects.
- [install.sh]: **`read -r` in a pipe-to-sh context** — The script is advertised as `curl ... | sh`, but when piped, stdin is the curl output, not the terminal. The `read -r REPLY` for the pipx confirmation prompt will read from the pipe (which is empty/EOF), silently falling through to the `*` case and installing pipx without actual user confirmation. This defeats the stated "with user confirmation" requirement in FR-4.3. You need `read -r REPLY < /dev/tty` or detect non-interactive mode and default to a safe behavior.
- [tests/test_ci_workflows.py]: **Testing YAML structure is cargo-cult testing** — You wrote 117 lines of tests that verify a YAML file has certain keys. These tests tell you nothing useful — if someone breaks the workflow, these tests still pass because they only check structure, not behavior. The only real validation is running the workflow. At least this is harmless, but don't confuse it with real test coverage.
- [src/colonyos/doctor.py]: **Version line always reports True** — The doctor check for ColonyOS version is hardcoded `True`. Every doctor check should test something meaningful. What if the version is `0.0.0.dev0` (the fallback)? That's a degraded state worth flagging. Check for the fallback and report it.
- [install.sh:107]: **`$PYTHON -m pip install --user pipx`** — On many modern distros (Debian 12+, Ubuntu 23.04+), `pip install --user` fails with the `externally-managed-environment` error (PEP 668). The script doesn't handle this. The fallback to `pip install --user colonyos` has the same problem on line 654. This will blow up on exactly the Linux users you're trying to help.

SYNTHESIS:
The core CI and release workflow pieces are solid — the `ci.yml` is clean, the release pipeline structure (test → build → publish → release) is correct, the `setuptools-scm` integration is properly done, and the version deduplication works. The install script's dry-run mode is a nice touch. But the Homebrew formula is shipped in a non-functional state with a placeholder SHA and hardcoded version URL, and the release workflow has no step to update it — making the entire FR-5 dead on arrival. The install script has a real bug where `read` won't work in the pipe-to-sh context it's designed for, and pip-based installation will fail on modern externally-managed Python environments. Fix the install script stdin bug, either make the Homebrew formula actually work end-to-end (including auto-update in the release workflow) or remove it entirely, and deduplicate the test job between CI and release workflows. Don't ship broken things and call them done.
