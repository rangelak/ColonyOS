FINDINGS:
- [branch scope]: The branch contains 6 commits but only 1 (70f2f87) implements the resume feature from the PRD. The other 5 commits introduce entirely unrelated features: a CEO/auto command, redesigned parallel review system, fix loop, decision gate, and read-only tool restrictions. This is severe scope creep â€” 5 unrelated features bundled with the one feature the PRD requested.
- [src/colonyos/cli.py]: Importing private functions `_load_run_log` and `_validate_resume_preconditions` from `orchestrator.py` breaks encapsulation. These should be exposed via a public `load_and_validate_for_resume()` helper or the underscore prefix should be dropped since they're now part of the module's public API.
- [src/colonyos/orchestrator.py `resume_from` dict]: The `resume_from` parameter is typed as `dict | None` â€” an untyped dictionary with 5 expected keys (`log`, `branch_name`, `prd_rel`, `task_rel`, `last_successful_phase`). This is a "stringly-typed" interface that will break silently on typos. Should be a `@dataclass` (e.g., `ResumeState`) with proper type annotations. Prompts are programs, and so are data contracts â€” treat them with the same rigor.
- [src/colonyos/orchestrator.py `_SKIP_MAP`]: The decision â†’ skip set is `{"plan", "implement", "review"}` but the `_compute_next_phase("decision")` returns `"deliver"`. The skip map for `"decision"` omits the decision gate itself from the skip set, which works because the decision gate is inside the review block, but this implicit coupling is fragile. If someone reorders phases or adds a new one, this will silently break.
- [src/colonyos/orchestrator.py]: The `run()` function has grown to ~250+ lines with nested control flow for plan/implement/review-fix-loop/decision/deliver, plus resume skip guards interleaved throughout. The resume feature adds more conditional branches to an already complex function. A phase-runner abstraction (list of phase callables, iterate and skip) would make the resume logic trivially correct by construction rather than correct by careful placement of `if "phase" in skip_phases` guards.
- [src/colonyos/models.py]: `RunLog` is `frozen=True` but the resume logic does `log.status = RunStatus.RUNNING` and mutates `log.phases`. This works because dataclasses' `frozen=True` only prevents `__setattr__` at the Python level, and `object.__setattr__` or list mutation bypasses it â€” but it's a correctness smell. The `mark_finished()` method uses `object.__setattr__` to work around frozen, which is fine for a single method, but the resume path does `log.status = RunStatus.RUNNING` which should fail on a frozen dataclass unless it's also using `object.__setattr__`. Need to verify this doesn't raise `FrozenInstanceError` at runtime.
- [tests/test_orchestrator.py]: The resume E2E test (`test_log_continuity_preserves_original_phases`) verifies log continuity well, but there's no test for the failure mode where `resume_from` dict has a missing key (e.g., no `"log"` key). Since it's an untyped dict, a `KeyError` at runtime is the current behavior â€” a `ResumeState` dataclass would catch this at construction time.
- [src/colonyos/cli.py `auto` command]: Entirely out of scope for this PRD. The `colonyos auto` command with CEO phases, proposal generation, `--loop`, `--propose-only`, and `--no-confirm` is a separate feature that should have its own PRD, review, and branch.
