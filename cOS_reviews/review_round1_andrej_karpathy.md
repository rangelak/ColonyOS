FINDINGS:
- [src/colonyos/orchestrator.py:294]: **Bug: redundant review loops when `enable_fix=False`**. When `enable_fix=False` and reviewers request changes, there is no `break` or `else` clause after the `if enable_fix and iteration < config.max_fix_iterations:` block. The loop will re-run the same reviews `max_fix_iterations + 1` times against unchanged code — burning budget on identical, redundant LLM calls. This is the kind of stochastic-output failure mode that silently wastes money. Fix: add `else: break` after the fix block (line ~330) to bail out of the loop when fix is disabled and changes are requested.
- [src/colonyos/orchestrator.py:372]: **Code smell: displaced module-level import**. `from colonyos.ui import NullUI, PhaseUI` is placed at line 372, wedged between the end of `run_review_loop()` and the definition of `_touch_heartbeat()`. This works at runtime (Python executes module-level statements top-to-bottom before any function is called), but it violates PEP 8 import ordering and is confusing to read. It should be moved to the top-of-file import block (line ~16). This appears to be a merge artifact from inserting the new code above the original import location.
- [src/colonyos/cli.py:377-390]: **Fragile reviewer-verdict matching**. The `_print_review_summary` logic in the `review` command matches REVIEW phase results to persona names by counting `len(reviewer_verdicts) < len(reviewers)` and using modulo indexing. This is brittle — if `run_review_loop` runs multiple rounds (as it will with the loop bug above), the phase results accumulate across rounds, and the matching logic silently drops all but the first N results. A more robust approach would be to tag each `PhaseResult` with the persona name, or filter to the last round's results.
- [src/colonyos/orchestrator.py:294-306]: **Subtle logic gap: standalone fix with `prd_rel` but no `task_rel`**. When `prd_rel` is provided but `task_rel` is None (the standalone review case with `--prd`), the code takes the `else` branch and uses `_build_standalone_fix_prompt()`, which doesn't reference the PRD at all. The condition should be `if prd_rel and task_rel:` → `elif prd_rel:` (use PRD-only fix prompt) → `else:` (standalone). Currently the PRD context is silently discarded in fix iterations when `--prd` is used without a task file, which is exactly the standalone review use case.
- [src/colonyos/instructions/review_standalone.md:34]: **Prompt tells reviewer to run `git diff {base_branch}...HEAD`** but the reviewer is operating on a named branch, not necessarily `HEAD`. If the agent's working directory isn't checked out to the target branch, this instruction produces the wrong diff. Should be `git diff {base_branch}...{branch_name}` for correctness, matching the decision_standalone.md template which correctly uses `{base_branch}...{branch_name}`.

The prompt templates are well-structured with consistent output formats, which is critical when you're parsing stochastic outputs. The displaced import and fragile verdict matching are minor but indicate the extraction was done a bit hastily. Fix the loop break logic and the PRD-in-fix-mode issue and this is ready to ship.
