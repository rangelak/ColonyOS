FINDINGS:
- [src/colonyos/ci.py]: The `poll_pr_checks` completion detection logic (line 289-292) uses an overly permissive OR condition — a check with `state=""` AND `conclusion=""` would pass the `all_done` filter, potentially causing premature return on checks that haven't started yet. However, the `and checks` guard and the fact that `gh pr checks` returns actual states in practice mitigates this. Worth a follow-up tightening but not a blocker.
- [src/colonyos/cli.py, line 1526]: The `--wait` polling after the last attempt has a subtle logic split: `if wait and attempt < max_retries` vs `elif wait and attempt >= max_retries`. When `max_retries=1`, the first attempt is also the last, so the `elif` branch fires. This works correctly but the duplicated poll-check-return logic could be a single block. Minor code quality nit.
- [src/colonyos/cli.py, line 1503]: `hasattr(config, "get_model")` guard is unnecessary — `ColonyConfig` always has `get_model`. This is defensive coding against a condition that can never occur. Harmless but noisy.
- [src/colonyos/orchestrator.py, line 1338]: `import subprocess as _sp` inside the function body is inconsistent with the module-level imports elsewhere in orchestrator.py. Similarly `import re as _re` at line 1270. These deferred imports add no real benefit since `subprocess` and `re` are stdlib modules already used at module level.
- [src/colonyos/instructions/ci_fix.md]: The prompt template is well-structured with clear constraints (no refactoring, no features, no PR description changes). Good treatment of prompts as programs. The structured step-by-step process matches how I'd want an agent to approach CI debugging. One observation: the template doesn't explicitly instruct the agent to `git add` and `git commit` — it says "Commit all fixes" but the agent might not know it needs to stage first. In practice, Claude Code handles this fine, so not a real issue.
- [src/colonyos/sanitize.py]: Secret patterns are solid for v1. The `Bearer \S+` pattern will catch `Bearer [REDACTED]` on re-application, but since sanitization is applied once per log fetch, this is fine. The high-entropy base64 pattern requires an adjacent keyword, which is the right trade-off — avoids false positives on legitimate base64 in test fixtures while catching real leaks.
- [src/colonyos/orchestrator.py, line 1715-1716]: The orchestrator integration is clean — gated by `config.ci_fix.enabled and config.phases.deliver`, runs after deliver, and the function returns early if PR number can't be extracted. FR21 (budget enforcement) relies on the existing `per_phase` budget passed to each agent invocation, which is correct but means CI fix retries each get a full `per_phase` budget rather than sharing from `per_run`. The PRD says "CI fix cost counts against `budget.per_run`" but the implementation doesn't enforce the aggregate run budget — it only enforces per-phase. This is a gap but matches how all other phases work today, so it's consistent.
