# Principal Systems Engineer Review — Pipeline Lifecycle Hooks

**Branch**: `colonyos/recovery-24cd295dcb`
**PRD**: `cOS_prds/20260402_071300_prd_...hooks...`
**Date**: 2026-04-02

## Review

VERDICT: request-changes

FINDINGS:
- [src/colonyos/orchestrator.py]: `on_failure` hooks are only triggered when a *hook* fails (via `_hooks_at` returning False), not when the pipeline itself fails (e.g., plan phase returns failure, implement phase crashes). The PRD (FR-3.1) explicitly requires `on_failure` when `_fail_run_log()` is called. There are ~21 non-hook `_fail_run_log` call sites that never trigger `on_failure` hooks. This is the primary gap — user story 4 ("notified when a pipeline run fails") is broken for the most common failure mode.
- [src/colonyos/orchestrator.py]: `_zip_results_with_configs` reaches into `hook_runner._hooks` (a private attribute). This is a brittle coupling — the HookRunner should expose a public method or the blocking check should be done inside HookRunner itself. `run_hooks` already stops on blocking failure; the caller could simply check `results[-1].success` to detect the stop-early case.
- [src/colonyos/orchestrator.py]: The `post_review` hook fires even when the review phase is skipped (via `skip_phases` or `config.phases.review` being False). Line 5017+ places `post_review` outside the review guard blocks, meaning it fires unconditionally. Same issue: if review is skipped, `pre_review` is never called but `post_review` is.
- [src/colonyos/orchestrator.py]: The deliver phase with `_drain_hook_output()` is called right before `return run_phase_sync(...)` — but if the deliver phase succeeds via the fallback `_do_pr_and_deliver_ui()` path (line ~5050+), hook output is never drained or injected into that path.
- [src/colonyos/hooks.py]: `shell=True` with user-provided commands is noted as an open question in the PRD but was shipped without comment. From a reliability perspective: this is fine for V1 (users own the config), but the choice should be documented in a code comment explaining the trade-off.
- [src/colonyos/hooks.py]: `_should_scrub_key` scrubs any env var containing "KEY", "TOKEN", "SECRET" as substrings. The `_SAFE_ENV_EXACT` allowlist is a good start but will inevitably hit false positives (e.g., `KEYBOARD_TYPE`, `npm_config_access_token`, `DOCKER_CONTENT_TRUST_REPOSITORY_KEY`). This is a landmine for user toolchains — a hook running `docker push` or `npm publish` will silently fail because credentials were scrubbed. Consider logging scrubbed variable names at DEBUG level so users can diagnose.
- [src/colonyos/hooks.py]: `text=True` in `subprocess.run` means non-UTF8 output causes `UnicodeDecodeError`, caught by the bare `except Exception` handler and reported as exit_code=-1. The test acknowledges this. Using `encoding="utf-8", errors="replace"` would be more robust and give users partial output instead of nothing.
- [src/colonyos/sanitize.py]: `sanitize_hook_output` calls `sanitize_display_text` then `sanitize_ci_logs`, but the PRD (FR-4.1) specifies the chain as `sanitize_display_text` -> `sanitize_ci_logs` -> `sanitize_untrusted_content`. The implementation omits the `sanitize_untrusted_content` pass. This means XML/injection-style content in hook output may not be fully stripped (though `sanitize_ci_logs` does strip XML tags, so the practical risk is low).
- [src/colonyos/cli.py]: The `hooks test` command uses `HookRunner.run_hooks` which stops on the first blocking hook failure. This means `--all` won't test all hooks if any blocking hook fails early. For a *test* command, the user likely wants to see all results. Consider using `run_on_failure`-style best-effort execution or a test-specific mode.
- [tests/test_hooks.py]: Tests modify `os.environ` directly with manual cleanup in `try/finally`. Using `monkeypatch.setenv` from pytest would be safer and more idiomatic — if an assertion fails before `finally`, the cleanup still runs, but `monkeypatch` is designed exactly for this.

SYNTHESIS:
The implementation is structurally sound — the `HookRunner` class is well-isolated, testable, and the config parsing follows existing patterns cleanly. The 757 passing tests (including 26 dedicated hook tests + orchestrator wiring tests) give good confidence in the happy path. However, there is one critical gap: `on_failure` hooks are not wired into the general pipeline failure path (`_fail_run_log`), which means the most important use case — "notify me when my pipeline fails" — only works when a *hook* causes the failure, not when a *phase* fails. This is a semantic bug that would confuse users at 3am. The `post_review` hook firing when review is skipped is a correctness issue that could gate deployments incorrectly. The private attribute access in `_zip_results_with_configs` is a code quality concern. I recommend fixing the `on_failure` wiring and the `post_review` placement before merge; the other items are improvements that can be addressed in follow-ups.
